Overview of the Penpie Reentrancy Attack

Before diving into the solution, let’s briefly recap what happened with Penpie. The protocol experienced a reentrancy attack, where a malicious actor exploited the harvestBatchMarketRewards() function. This function, like other functions vulnerable to reentrancy, allowed the attacker to repeatedly call it before the original call was completed, resulting in unauthorized withdrawals of funds.

The absence of reentrancy guards in the code allowed this exploit to occur. While traditional testing techniques might have missed such a subtle vulnerability, mutation testing could have flagged it. Let’s understand how.

What is Mutation Testing?

Mutation testing is a powerful method used to measure the effectiveness of test cases. It involves introducing small, deliberate changes or mutations to the code. These mutations simulate bugs or vulnerabilities that might occur in the contract’s codebase.

The mutated code, known as the mutant, is then tested using the existing test suite. If the test suite fails to catch the change, it indicates that the tests are insufficient, and the coverage is incomplete. This prompts developers to write additional tests or modify the existing ones to catch such issues.

Mutation testing helps ensure that your codebase is well-tested, and critical security measures are in place to prevent attacks like the one on Penpie.

How Mutation Testing Works

1. Apply Mutation: Small changes are made to the smart contract, such as removing a line of code or altering conditions within functions.

2. Run Tests: The mutated contract is then run against the test suite.

3. Evaluate Results:

• If the tests pass despite the mutation, it means that the tests are incomplete. Developers need to write more tests to cover the untested behavior.

• If the tests fail, it indicates that the tests are catching the issue, resulting in a secure contract.

Applying Mutation Testing to Smart Contracts

When applied to smart contracts, mutation testing is particularly effective in identifying vulnerabilities like reentrancy issues. By simulating mutations that alter the behavior of state-changing functions, developers can evaluate whether their contracts are safe under all circumstances.

Example: Testing Deposit Functionality

Consider a typical deposit function in a smart contract:

function deposit() external payable {
 require(msg.value > 0, “Deposit must be positive”);
 balances[msg.sender] += msg.value; totalDeposits += msg.value; 
}

Now, what happens if the mutation testing tool removes the require statement that checks for a positive deposit value? If the test suite does not detect this change and passes the tests, it indicates that this scenario has not been thoroughly tested.
